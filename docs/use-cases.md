# Use Cases

웹 서비스 보안 진단 시스템의 주요 사용자 시나리오를 정리한 문서입니다.

## UC-01: 검사 대상 등록

| 항목 | 내용 |
| --- | --- |
| 유즈케이스 ID | UC-01 |
| 유즈케이스명 | 검사 대상 등록 |
| 액터(Actor) | 사용자(User) |
| 관련 액터 | 없음 |
| 목적(Goal) | 사용자가 진단할 웹 서비스의 기본 정보를 입력한다. |
| 사전조건(Precondition) | 없음 |
| 사후조건(Postcondition) | 검사 대상(Target)이 시스템에 등록된다. |

### 기본 흐름

1. 시스템은 기본 URL 입력 화면을 제공한다.
2. 사용자는 대상 웹 서비스의 기본 URL을 입력한다.
3. 시스템은 입력값 형식을 검증한다.
4. 시스템은 검사 대상 정보를 등록한다.
5. 사용자는 등록된 검사 대상을 확인한다.

### 대체 흐름

#### A1. URL 형식 오류

1. 시스템은 올바르지 않은 URL 형식임을 알린다.
2. 사용자는 URL을 다시 입력한다.

## UC-02: 검사 범위 설정

| 항목 | 내용 |
| --- | --- |
| 유즈케이스 ID | UC-02 |
| 유즈케이스명 | 검사 범위 설정 |
| 액터(Actor) | 사용자(User) |
| 관련 액터 | 없음 |
| 목적(Goal) | 사용자가 검사할 엔드포인트를 설정한다. |
| 사전조건(Precondition) | 검사 대상이 등록되어 있어야 한다. |
| 사후조건(Postcondition) | 검사 엔드포인트가 저장된다. |

### 기본 흐름

1. 사용자는 검사 대상 엔드포인트를 직접 입력한다.
2. 시스템은 검사 설정 화면을 제공한다.
3. 시스템은 엔드포인트 정보를 저장한다.

### 대체 흐름

#### A1. 엔드포인트 미입력

1. 시스템은 최소 1개 이상의 엔드포인트 입력을 요구한다.

#### A2. 엔드포인트 확인

1. 시스템은 HTTP 응답이 `200 OK`인지 확인한다.
2. 응답이 정상적이지 않으면 사용자에게 재입력을 요구한다.

## UC-03: 서버 스택/버전 입력 및 식별

| 항목 | 내용 |
| --- | --- |
| 유즈케이스 ID | UC-03 |
| 유즈케이스명 | 서버 스택/버전 입력 및 식별 |
| 액터(Actor) | 사용자(User) |
| 관련 액터 | 대상 웹 서비스, CVE DB |
| 목적(Goal) | 사용자가 서버/프레임워크 정보를 입력하거나 시스템이 이를 식별하여 CVE 매핑에 활용한다. |
| 사전조건(Precondition) | 검사 대상과 범위가 설정되어 있어야 한다. |
| 사후조건(Postcondition) | 스택 정보가 저장되고 CVE 매핑 기준이 준비된다. |

### 기본 흐름

1. 사용자는 서버 스택 정보를 입력한다.
   예: `React`, `Nginx`, `Spring Boot`
2. 시스템은 입력값을 저장한다.
3. 필요 시 시스템은 헤더, 응답 정보, 정적 파일 특성 등을 통해 버전을 추정한다.
4. 시스템은 스택 정보와 CVE DB를 연결할 준비를 한다.

### 대체 흐름

#### A1. 버전 식별 불가

1. 시스템은 버전 미확인 상태로 저장한다.
2. 이후 일반 진단 항목 중심으로 검사를 진행할 수 있다.

## UC-04: 인증 정보 설정 및 로그인 상태 확보

| 항목 | 내용 |
| --- | --- |
| 유즈케이스 ID | UC-04 |
| 유즈케이스명 | 인증 정보 설정 및 로그인 상태 확보 |
| 액터(Actor) | 사용자(User) |
| 관련 액터 | 대상 웹 서비스 |
| 목적(Goal) | 사용자가 로그인 후 접근 가능한 관리자 페이지나 사용자 전용 영역에 대한 인증 정보를 설정하고, 시스템이 인증 상태를 확보한다. |
| 사전조건(Precondition) | 검사 대상과 범위가 설정되어 있어야 한다. |
| 사후조건(Postcondition) | 로그인 세션 또는 인증 정보가 확보되어 이후 인증 상태에서 스캔을 수행할 수 있다. |

### 기본 흐름

1. 사용자는 "로그인 후 검사" 옵션을 선택한다.
2. 시스템은 로그인 URL, HTTP Method, ID/PW 또는 세션 토큰 입력을 요청한다.
3. 사용자는 인증 정보를 입력한다.
4. 시스템은 대상 웹 서비스에 로그인 요청을 수행한다.
5. 로그인 성공 시 시스템은 세션, 쿠키 또는 토큰을 유지한다.
6. 시스템은 인증 상태를 저장하여 이후 스캔에 활용한다.

### 대체 흐름

#### A1. 로그인 실패

1. 시스템은 인증 실패 메시지를 출력한다.
2. 사용자는 인증 정보를 다시 입력한다.

#### A2. MFA 또는 캡차 존재

1. 시스템은 자동 로그인이 제한됨을 표시한다.
2. 사용자는 수동 세션 값을 입력할 수 있다.

#### A3. 세션 만료

1. 시스템은 재인증이 필요함을 표시한다.
2. 해당 인증 정보는 다시 입력하거나 갱신해야 한다.

## UC-05: CVE 및 진단 항목 기반 스캔 실행

| 항목 | 내용 |
| --- | --- |
| 유즈케이스 ID | UC-05 |
| 유즈케이스명 | 보안 스캔 실행 |
| 액터(Actor) | 사용자(User) |
| 관련 액터 | 대상 웹 서비스, CVE DB |
| 목적(Goal) | 입력된 대상과 범위를 기준으로 CVE 및 보안 진단 항목 검사를 수행한다. |
| 사전조건(Precondition) | 대상, 범위, 스택 정보, 인증 정보가 필요한 경우 모두 설정되어 있어야 한다. |
| 사후조건(Postcondition) | 스캔 결과와 발견된 취약점이 저장된다. |

### 기본 흐름

1. 사용자는 스캔 실행 버튼을 누른다.
2. 시스템은 검사 설정 정보를 불러온다.
3. 시스템은 엔드포인트별 요청을 수행한다.
4. 시스템은 서버 스택/버전과 CVE DB를 비교한다.
5. 시스템은 진단 항목을 검사한다.
   진단 예시: `SQL Injection`, `XSS`, 보안 헤더, 인증/인가 관련 기본 점검 항목
6. 시스템은 취약점 여부, 근거, 심각도를 정리한다.
7. 시스템은 결과를 저장한다.
8. 시스템은 사용자에게 결과 요약을 표시한다.

### 대체 흐름

#### A1. 특정 엔드포인트 응답 없음

1. 시스템은 해당 항목을 실패 또는 시간 초과로 기록한다.
2. 나머지 검사는 계속 수행한다.

#### A2. 스택 버전 미확인

1. 시스템은 CVE 매핑 대신 일반 진단 항목 검사만 수행한다.

#### A3. 인증 필요한 페이지 접근 실패

1. 시스템은 해당 영역을 "인증 실패로 검사 불가"로 기록한다.

## UC-06: 결과 조회 및 리포트 출력

| 항목 | 내용 |
| --- | --- |
| 유즈케이스 ID | UC-06 |
| 유즈케이스명 | 결과 조회 및 리포트 출력 |
| 액터(Actor) | 사용자(User) |
| 관련 액터 | 없음 |
| 목적(Goal) | 사용자가 스캔 결과를 확인하고 리포트를 출력한다. |
| 사전조건(Precondition) | 스캔 결과가 존재해야 한다. |
| 사후조건(Postcondition) | 사용자는 취약점 결과와 권고사항을 확인할 수 있다. |

### 기본 흐름

1. 시스템은 발견된 취약점 목록을 보여준다.
2. 사용자는 취약점 상세 정보(CVE ID, 설명, 심각도, 영향받는 경로, 재현 근거)를 확인한다.
3. 시스템은 화면용 리포트를 제공한다.

### 대체 흐름

#### A1. 취약점 없음

1. 시스템은 "발견된 취약점 없음"을 표시한다.

#### A2. 일부 증거 누락

1. 시스템은 재현 근거가 부족한 항목을 별도로 표시한다.
2. 시스템은 검사 대상과 일치하는 CVE가 없거나 근거가 불충분함을 함께 안내한다.
